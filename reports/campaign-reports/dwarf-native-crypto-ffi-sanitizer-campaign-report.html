DWARF — Native-Crypto FFI Sanitizer Fuzzing

Memory-safety fuzzing of cardano-node's C crypto (libsodium / secp256k1 / blst) under AddressSanitizer · build host, 2026-06-22 · the FU10 surface
cardano-node is written in Haskell (memory-safe), so the only place a memory-corruption bug can hide is the C-FFI crypto. This campaign fuzzed all three native crypto libraries cardano links — the IOG libsodium fork (Ed25519, Praos VRF, blake2b), secp256k1 (Plutus ECDSA/Schnorr), and blst (BLS12-381, Plutus builtins) — directly under AddressSanitizer + libFuzzer + SanitizerCoverage, each seeded with valid signatures/keys/points so the deep verify/curve-math paths are exercised, not just early rejection. ~1.51 billion executions across 7 attacker-facing entrypoints (1 hour each) — zero crashes, zero ASan events, zero memory-safety findings. This closes the one fuzzing surface (memory corruption) that the decoder/ledger/Antithesis work could not reach.

Method

The IOG libsodium, secp256k1, and blst sources on the build host were rebuilt with clang 18 + ASan + SanitizerCoverage (-fsanitize=address,fuzzer-no-link -fsanitize-coverage=inline-8bit-counters,trace-cmp,trace-div,trace-gep). libsodium/secp256k1 use --disable-asm for clean instrumentation; for blst the C (server.c) is instrumented while its hand-written field-arithmetic assembly is assembled normally (standard blst-fuzzing approach — the memory surface is the C deserialize/parse logic). One libFuzzer harness per attacker-facing entrypoint; for verify/parse paths the harness also fuzzes the upstream deserializers (pubkey/signature/point parse) since those touch attacker input first. Each harness was seeded with 80 valid inputs generated from the real libraries, then run 1 hour with leak-detection disabled (so only genuine memory corruption — use-after-free, buffer overflow, OOB, bad free — trips the run).

What this proves (and the honest scope)

Proven: across ~1.5B adversarial executions, every native crypto primitive cardano-node links — Ed25519 verify, Praos VRF verify, blake2b, Plutus secp256k1 ECDSA/Schnorr (parse+verify), and Plutus BLS12-381 G1/G2 (uncompress/deserialize + subgroup check) — exhibited no memory-safety violation (no use-after-free, buffer overflow, OOB read/write, or bad free) under AddressSanitizer. This is the memory-corruption class that Haskell cannot have but a C-FFI dependency could.
  • Scope: all three actual C libraries cardano links (IOG libsodium fork, secp256k1, blst) — the complete native memory surface. Coverage-guided generational fuzzing (libFuzzer + SanCov).
  • Finds: memory-corruption + sanitizer-detectable UB. (Functional correctness of the crypto math is upstream's formal/test remit.)
  • Not in scope here: KES (key-evolving signatures — a Haskell composition over the covered Ed25519/blake2b primitives, not a distinct C entrypoint); blst pairing/miller-loop and hash-to-curve (the uncompress/subgroup parse is the primary attacker surface; pairing is a candidate follow-on harness); constant-time/side-channel properties (a different analysis).
Net: the FU10 "FFI sanitizers" surface — the last untouched fuzzing class, and the only one where RCE/consensus-split-class memory bugs could live — is now exercised at volume across all three crypto libs with a clean result.

Reproducibility & bundled evidence (ffi-crypto-fuzz-evidence.tar.gz)

  • harness/ — the 7 libFuzzer harnesses (ed25519/blake2b/vrf03 · secp ecdsa/schnorr · blst g1/g2)
  • gen/ — seed-corpus generators (gen_sodium.c, gen_secp.c, gen_blst.c — valid sig/key/proof/point material)
  • corpus/ — seed corpora (80 valid inputs per harness; random test bytes, no real keys)
  • logs/ — full libFuzzer run logs + campaign window
  • BUILD.md, configure*.log, make-*.log, blst-build.log — exact instrumented-build provenance (clang flags, configure options, blst C+asm split)

Determinism: harnesses + corpora rebuild from the cardano c-deps source with the documented flags; libFuzzer is corpus-driven and re-runnable. The large instrumented archives are not bundled — rebuilt from source per BUILD.md.
