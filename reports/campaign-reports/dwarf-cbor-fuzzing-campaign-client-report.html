DWARF — Cardano Node CBOR Fuzzing Campaign

cardano-node decoder-robustness assessment (Antithesis + local) · generated 2026-06-16 15:36Z
DWARF subjected a live cardano-node to large volumes of adversarial CBOR across the block, transaction, and auxiliary-data decoders — over the real Cardano node-to-node protocols (block-fetch and tx-submission), on the Antithesis deterministic simulator and in multi-hour local soaks. Two complementary fuzzing layers were exercised: (1) hostile-but-valid structural mutation (well-formed CBOR with corrupted structure/types/values) and (2) outright malformed / invalid CBOR bytes (truncated, bit-flipped, oversized-length, deeply-nested, raw garbage). Across every run the node's decoders handled the input without crashing, hanging, or violating consensus — valid-but-hostile payloads were rejected at validation and malformed bytes were rejected with a clean DecoderFailure. A positive robustness result on the exercised surface.

What was tested

Assess the robustness of cardano-node's CBOR deserialization under adversarial input. The decoder is a high-value attack surface: it processes untrusted bytes off the network before any validation. The campaign targets the block decoder (block-fetch), the transaction and auxiliary-data decoders (tx-submission), and the block-header decoder (chain-sync), under both structural mutation and byte-level malformation.

Method

A purpose-built Haskell adversary (dwarf-adversary) joins the testnet as a genuine node-to-node peer (chain-sync, block-fetch, tx-submission, keep-alive), bootstraps a valid chain, then serves payloads the node decodes with its real production decoder. Two mutation layers (--mutation-level): struct — mutate the parsed CBOR Term (field drop, type confusion, length/nesting corruption) and re-encode to valid CBOR with hostile structure; and bytes — corrupt the serialized bytes (truncate / bit-flip / oversize-length 2^64-1 / deep-nest / garbage) into malformed CBOR, exercising the deserializer's error handling and resource bounds. For block-fetch the node is eclipsed by topology so its only block source is the adversary. The base seed is drawn fresh per Antithesis timeline (per-timeline exploration of the mutation space). The node's own health assertions (no crashes / errors / consensus violations) are the bug oracle: rejecting bad input is correct behaviour; a crash would be a finding.

Findings

Result — cardano-node is robust to adversarial CBOR on every surface tested. Across 8 live Antithesis runs (each serving its target decoder; all serve-assertions passed) and two multi-hour local soaks — spanning the block, transaction, auxiliary-data, and block-header decoders, under both hostile-but-valid structural mutation and outright malformed/invalid CBOR — the node decoded every adversarial payload without a single crash, hang, node error, critical log, or consensus violation. Valid-but-hostile payloads were rejected at validation; malformed bytes were rejected with a clean DecoderFailure (including the oversize-length 264-1 DoS vector, which was rejected without allocating).

Scale. The block decoder was exercised by two ~3h Antithesis campaigns (~150 simulated test-hours each, structural and malformed) plus 8h local soaks serving ~169,000 mutated blocks apiece.

Non-node observations (investigated; none are node defects):
  • p2 producer exit + "no critical logs" (1h structural run): a stock producer, not the eclipsed node-under-test. Ruled out as a DWARF/node issue — not reproducible locally (the node logged 0 Error/Critical over 36k+ mutated blocks), the adversary has no connection path to p2, and it did not recur in the 3h campaign. A producer-mesh condition specific to the Antithesis simulator.
  • chain-sync flap driver (chain-sync-client/parallel_driver_flap.sh, 3h malformed run): a bundled harness test-template command exited non-zero. Not a node assertion — every node-under-test health check passed in that run. A composer-driver artifact, plausibly perturbed by byte-mode's aggressive peer disconnects.
  • "The Antithesis Fault Injector was started" fails in every run by design — all runs used --no-faults to isolate the decoder signal.
Known limitations. (1) The certificate CBOR shape is built but not yet meaningfully fuzzed live: the testnet's generator emits only payment transactions, so certificate-targeted mutation falls back to whole-tx — genuine certificate fuzzing needs cert-bearing transactions. (2) Local soaks plateau after ~4h (a DWARF-adversary longevity disconnect under one sustained eclipse connection — a test-harness limitation, not a node defect; the node stayed healthy the full duration in both soaks). (3) A clean result is strong evidence of robustness on the exercised surface within the run budget, not a proof of total absence of bugs. Malformed-CBOR coverage rounded out: dedicated byte-level runs on the transaction decoder (tx-submission) and the block-header decoder (chain-sync) both passed — the node decoded the malformed input and rejected it with no crash, no node errors, and no criticals. Malformed-byte fuzzing is now proven across block, transaction, and header decoders (matching the structural-fuzzing breadth); only the certificate shape remains untested in either family. Reject-oracle (FU1) — node correctly REJECTS, never accepts: beyond not crashing, a local reject-oracle confirmed the node actively rejects adversarial CBOR and never adopts it. Over a struct soak the node-under-test logged 2,445 DecoderFailure rejections against the adversary's stream and adopted ZERO of the 165,560 mutated blocks its sole (eclipsed) peer served — 0 critical logs, 0 restarts. So the result is not merely crash-robustness: the node decodes adversarial input, rejects what it should, and accepts none of it. (Note: structural mutations on blocks are rejected at the decoder, not the validator — exercising the deeper validation layer needs a decodable-but-semantically-invalid mutation mode, tracked as a follow-up.) Validation-layer probe (FU3b, image 0.14.0): a new semantic mutation mode perturbs leaf values while preserving CBOR type+length, so blocks DECODE cleanly but are semantically invalid — reaching the node's validation layer (not just the decoder). At full mutation rate the node rejected every served block at header-body consistency validation (BlockFetchProtocolFailureWrongBlock) and adopted none, no crash. (Depth note: this exercises header-body consistency; reaching the deeper ledger/consensus rules requires body-hash-consistent mutation — a tracked follow-up.)

  • Every run draws a fresh seed per Antithesis timeline and logs it; any timeline is reproduced locally with `--seed 0x<value>` via tools/sp3a_topology_eclipse_repro.sh (struct) or the same with `bytes` (malformed).
  • Results were read from each run's Antithesis triage report (per testRunId) on the amaru-cardano tenant; this report's figures were audited against that source.

Scope & honest limitations

  • CBOR shapes proven live: block-header (chain-sync), block (block-fetch), tx-body and auxiliary-data (tx-submission). Certificate is built but needs genuine cert-bearing txs (the testnet's tx-generator emits only payment txs, so cert targeting would fall back to whole-tx) — pending.
  • Mutation is single-point structural perturbation of the CBOR term (drop / type-confuse / corrupt length+nesting), driven by a per-timeline random seed. It is broad random structural fuzzing under Antithesis timeline exploration — not coverage-guided or symbolic.
  • Runs were executed with Antithesis fault injection disabled (--no-faults) to isolate the decoder signal; a faults-on pass can combine fuzzing with network/process chaos.
  • A clean result means no defect was found on the exercised surface within the run budget — it is strong evidence of robustness, not a proof of total absence of bugs.
  • The failures labelled "The Antithesis Fault Injector was started" (every run) and "No Antithesis errors" are expected artifacts of the --no-faults configuration and the Antithesis platform's own setup checks — not node defects. The asteria-game/* command failures and dwarf_base_header_obtained (a header-path assertion that does not apply in tx-submission runs) are likewise harness/mode artifacts.
  • Run scope: this report covers the 8 successful Antithesis runs of the current adversary (images 0.10.0-0.12.0) plus two local soaks. Three earlier tenant runs are intentionally excluded as they do not bear on the node-robustness result: two superseded bundles (commits 317ac21, b14a287) that failed at Antithesis ingestion due to a bundle-packaging/test-harness issue since fixed (they never reached the node), and one earlier-code run (commit fcb29e9, image 0.5.1) that Completed but predates the serve fix, so its fuzzing seam (dwarf_served_mutated_tx) was unfound — superseded by the confirmed tx-submission runs herein.
