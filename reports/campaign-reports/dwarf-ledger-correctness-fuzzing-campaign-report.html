Local deep-dive campaigns on cardano-node (build-host, 2026-06-16 → 06-17) · the "professional-grade fuzzing" follow-ups · generated 2026-06-22
These local campaigns extend the Antithesis results along the two axes a professional security assessment demands beyond "it didn't crash": (1) correctness — proving the node actively rejects adversarial input at the right layer (decoder and ledger), accepting none; and (2) credible volume — millions-to-billions of executions with execution-rate metrics. Across every campaign the node rejected what it should and accepted nothing it shouldn't, with zero crashes, zero critical logs, and zero uncaught exceptions. 1 — Reject-oracle: the node actively rejects, never adopts raw log + notes

The standard Antithesis oracle is node-health ("no crash/critical/consensus violation"). A silent validation bypass (node accepts a payload it should reject) wouldn't trip it. The reject-oracle adds the missing assertion: mutated input is decode-rejected and never adopted.

Struct/bytes block soak — node logged 2,445 DecoderFailure rejections and adopted 0 of 165,560 mutated blocks → proves active rejection, not merely "doesn't crash". notes

Topology-eclipse seam check (tools/reject_oracle.sh / fu1_struct.log) — eclipsed relay2, single network: raw log 
dwarf_served_mutated_block=65  relay2 VRFKeyBadProof=0  adversary RestartCount=0
OK: SP3a seam fired under TOPOLOGY eclipse on a single network (Antithesis-compatible).
Served mutated blocks were fetched and decode-rejected; node never adopted the adversary's chain (VRFKeyBadProof=0, RestartCount=0).

2 — Ledger-layer reach: tampered tx accepted to mempool, then ledger-rejected raw log

The deepest correctness probe: a witness-signature-flip tx (tx_body byte-identical, so the txid matches and the node accepts it into the mempool rather than decode-rejecting it) — does the node then run full ledger validation and reject it? 8-hour soak, image 0.17.0:

Result: across 328 iterations / 3,266 witness-tampered txs over 8h, every one reached relay2's mempool and was ledger-rejected (MempoolRejectedByLedger); mempoolAdd=0 the entire run (none ever accepted), relay2_crit=0, relay2_restart=0. The reject-oracle now spans the decoder layer AND the ledger/mempool layer — the node runs full ledger validation on adversarial input and correctly rejects it, adopting none. This closes the "correctness, not just crash-robustness" gap for the tx path. Raw: logs/witness_soak.log (mempoolAdd never exceeds 0 — verified across all 328 lines).

15,000,000 decodes across three mutation levels → 0 uncaught exceptions, 0 timeouts (≈11.3M clean-rejects + ≈3.7M decoded-ok). Deterministic via --seed, so any finding replays.

Extended runs notes: a 1h run did ~905M decodes and an 8h soak did ~7.2 BILLION decodes (bytes 2.71B + struct 3.65B + semantic 0.84B) → 0 exceptions, 0 timeouts; a corpus-diversity run over 17 distinct Conway decoder-branch seeds did ~225M decodes, 0 exceptions/timeouts. The 15M result set is persisted (quoted above + in tarball); the 905M / 7.2B / 225M totals are from the engineering note (those longer soaks logged only summary lines, not retained).

Honest scope: generational, not coverage-guided (GHC emits no SanitizerCoverage for AFL/libFuzzer — that gap is covered by the separate 8h native-SanCov campaign). Finds the DoS class (uncaught exceptions / non-termination); memory-corruption is a C-FFI surface (not in scope here).
