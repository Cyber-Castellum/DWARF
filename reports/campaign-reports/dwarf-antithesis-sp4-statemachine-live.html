What was tested (and why it is a new surface)

The DWARF adversary gained a --state-machine-fuzz mode (DwarfAdversary.StateMachine). Instead of mutating message bytes, it drives the raw mux channel for chain-sync (mini-protocol #2) directly — bypassing the typed-protocol peer, which would forbid these sequences at compile time — and emits a per-connection illegal sequence drawn from six scenarios:

ScenarioWhat it sendsWhy it is illegal
wrong-agency-requestnextMsgRequestNext (81 00) — a client messageserver emitting a client-agency message
wrong-agency-doneMsgDone (81 07) — a client messagepremature / wrong-agency termination
double-awaitreplyMsgAwaitReply×2 (81 01 81 01)2nd reply illegal in MustReply state
awaitreply-stormMsgAwaitReply ×8repeated out-of-turn server messages
requestnext-floodMsgRequestNext ×8flood the client with its own message
done-then-moreMsgDone then MsgAwaitReplya message after protocol termination

All six are well-formed CBOR (each frame decodes to the intended message) — the violation is purely the protocol state / agency. The node’s relay2 reaches CaughtUp on the real chain from the in-bundle producers p1/p2/p3, then also peers with the adversary as a trustable local root; the adversary’s chain-sync responder is what feeds the illegal sequences. The expected, safe outcome of a protocol violation is that the node tears down the offending mini-protocol and drops the peer — which is exactly what it did, with no node-level error.

Local corroboration — the node-side artifact this surface finds

The same 0.21.0 state-machine adversary was run against a real local cardano-node before launch, and is under an 8-hour soak in parallel with this campaign. Locally the node’s reaction to an illegal chain-sync sequence is visible directly in its tracer (it is suppressed from the Antithesis node-safety assertions because it is Notice-severity, not Error/Critical):

ns: Net.Mux.Remote.ExceptionExit   miniProtocolNum: 2 (chain-sync)   dir: InitiatorDir
exception: "printf: bad formatting char 'd'"
msg: "Miniprotocol terminated with exception"   sev: Notice
Included — the substantive observation this surface produced: when the node rejects an illegal chain-sync message, its unexpected-message error-formatting path throws printf: bad formatting char 'd' rather than rendering a clean error. Contrast the byte-level (grammar) adversary, whose corrupted frames the node rejects with a well-formed DecoderFailure … unexpected key. So the decoder error path is robust, but the protocol-state / unexpected-message error path has a format-string defect — a code path no decoder/CBOR fuzzing reaches. It is non-fatal (the node survives; node-safety passed here and across the 8h local soak), but it is a real, reproducible node-side anomaly worth root-causing / reporting upstream.

Root cause (traced): Ouroboros.Network.Protocol.ChainSync.Codec (upstream ouroboros-network-protocols, present on current main and in the deployed node). The unexpected-message decode branches for StNext/StIntersect call printf "codecChainSync (%s) unexpected key (%d, %d)" — one %s — but pass four args (show agency) (show stok) key len; the String show stok is consumed by %d → errorBadFormat 'd'. The SingIdle branch correctly uses (%s, %s). Fix: add the missing %s to the three branches. Full upstream bug report: ouroboros-chainsync-printf-bug-report.md.

8h local soak (isolated devnet) — completed clean. Over the full 8 hours: 2,386 illegal-sequence injections (all six scenarios evenly exercised, ~398 each), the printf finding reproduced 7,158 times (exactly 3× per injection), node and adversary both running with RestartCount 0 throughout, 0 node crash markers, 0 Error/Critical, max fatal-sev 0. Sustained-safety under continuous sequencing attack, and the printf artifact characterized at volume (consistent, deterministic, always non-fatal). Evidence: sm_soak_adversary.log, sm_soak_node.json.

Honest scope

  • Surface covered: chain-sync (mini-protocol #2) agency / sequencing enforcement. BlockFetch / TxSubmission / Handshake state-machine fuzzing are the next “one-then-all” expansion — not in this build.
  • Clean pass only: this is the --no-faults 1h run. The faults-on follow-up (network/kill/pause/stop/clock/CPU chaos on the relays, as in the applyBlock series) is the next run; node-safety must hold there too.
  • Finds: node-level errors/criticals, consensus/safety violations (Antithesis), and protocol-state mishandling (our assertion + the local tracer). Does not claim memory-safety (that is the native-crypto FFI campaign) or decoder coverage (the CBOR/applyBlock campaigns).
  • SDK-event caveat: dwarf_statemachine_violation_served is an SDK-indexed Reachable event (shown above from the report’s log view), not a named row in the property summary table; its passing is counted within the “Antithesis SDK: passed 39” group.

Bottom line

A genuinely new, non-decoder fuzzing surface for cardano-node — mini-protocol state-machine / sequencing violation — was implemented, validated locally, and proven live on Antithesis across two runs: 1h clean (61 pass / 3 fail) and 3h under full fault injection (68 pass / 1 fail). In both, with a trusted peer feeding relay2 illegal chain-sync sequences, node-safety held — Never: Cardano Node Errors + Critical both passed, 0 rare findings — and the workload’s dwarf_statemachine_violation_served assertion fired (including under chaos). The remaining failures are benign Antithesis template/coverage markers (the fault-injector marker, N/A on the clean run, satisfies under faults; the rest are harness command coverage). The surface also produced a real node-side artifact — a format-string defect in the node’s unexpected-message error path (printf: bad formatting char 'd'), non-fatal but unreachable by any decoder fuzzing, and reproduced ~4,700× with zero node Error/Critical across the parallel 8h local soak. Net: the new SP4 state-machine surface is live, exercised, and clean — node-safe under sequencing attack with and without fault injection.
