DWARF — Cardano Node applyBlock Ledger-Apply Fuzzing Campaign

cardano-node ledger-application robustness assessment (Antithesis + local) · generated 2026-06-22 03:30Z
DWARF drove a live cardano-node ledger-application path with large volumes of adversarial input, advancing beyond the CBOR decoder into the block-application / ledger-validation rules themselves. A new in-process workload (dwarf-decoder-fuzz --target applyblock, adversary image 0.19.0) decodes a mutated Conway transaction → assembles a block → runs BHeaderView → BBODY → LEDGERS → per-tx ledger rules over a genesis-initialised Conway NewEpochState, while a node-to-node adversary serves mutated transactions on the wire. The campaign ran on the Antithesis deterministic simulator across two runs — a 1-hour clean baseline and a 3-hour run with fault injection enabled — backed by an 8-hour local coverage-guided (SanCov) soak of the same applyBlock surface. Across every run the node applied/validated and rejected adversarial input without a single crash, hang, node error, critical log, or consensus violation — including while its relay nodes were fully exposed to network and process faults. A positive robustness result on the exercised surface.

What was tested

Prior DWARF campaigns proved the cardano-node decoders robust to adversarial CBOR (block, transaction, auxiliary-data, block-header). This campaign extends the probe one layer deeper — to ledger application: does the node correctly run and reject malformed/semantically-invalid input through the full block-body and per-transaction ledger rules, not merely decode-reject it. Two surfaces were exercised together:

  • In-process applyBlock fuzz — dwarf-decoder-fuzz --target applyblock (image 0.19.0): a tight in-process loop mutating a baked Conway corpus and running each result through decode → BHeaderView → BBODY → LEDGERS → per-tx ledger rules over a genesis-initialised NewEpochState (genesis + corpus baked into the public image). Bug oracle: any uncaught exception or non-termination fails an always(...) assertion.
  • Node-to-node transaction adversary (image 0.18.0): an eclipsed-peer tx-submission adversary serving structurally-mutated wire transactions to the node under test, with the node fully bootstrapped to CaughtUp via real producers.

Method

Bundle antithesis/cardano_node_dwarf at Cyber-Castellum/DWARF main@68c97a7b — the full-harness topology bundle (real producers bootstrap the node-under-test to GSM CaughtUp; the in-process decoder-fuzz-applyblock + tx adversary run as fault-excluded workloads). Public images ghcr.io/j-gainsec/dwarf-adversary:0.19.0 (applyblock + decoder-fuzz) and :0.18.0 (wire adversary). Mutation is seeded per-timeline from antithesis_random, so Antithesis explores a fresh seed on every timeline; the baked Conway genesis makes each apply deterministic and replayable. Both runs were launched via the on-chain Moog requester flow; the Antithesis oracle built and ran the bundle on the amaru-cardano tenant. The 1-hour run isolates the surface (--no-faults); the 3-hour run adds Antithesis network + process fault injection to the relay nodes.

Findings

Result — cardano-node is robust to adversarial input on the applyBlock ledger-apply surface. Across 2 live Antithesis runs (a 1-hour clean baseline and a 3-hour run with fault injection) and an 8-hour local coverage-guided soak, the node ran adversarial input through the full block-application and per-transaction ledger rules without a single crash, hang, node error, critical log, or consensus violation. No always(...) bug-oracle failed; no finding was classified rare (Antithesis's serious-defect class) in either run. The 3-hour result is the strongest: the relay nodes were fully fault-exposed (only the fuzz harness was excluded from faults) and still logged zero errors and zero criticals across the entire run.

Non-node observations (investigated; none are node defects):
  • Sometimes: dwarf_base_header_obtained not reached (both runs) — a reachability assertion for the chain-sync header path, which is legitimately not exercised when the adversary runs in tx-submission mode. Its sibling dwarf_base_tx_obtained fired (count 2). A mode-not-applicable miss, not a failure.
  • "All commands were started / run to completion at least once" — Antithesis test-template command-coverage markers (composer/workload harness), not node assertions.
  • "The Antithesis Fault Injector was started" — appears in the 1-hour run by design (it used --no-faults to isolate the surface).

Reproducibility

Both runs are pinned to Cyber-Castellum/DWARF main@68c97a7b, public images ghcr.io/j-gainsec/dwarf-adversary:0.19.0 and :0.18.0, on the amaru-cardano Antithesis tenant. testRunIds 0341f850…50ae (1h, no-faults) and 859ad183…311f (3h, faults-on). Per-input mutation is a deterministic function of the per-timeline seed (drawn from antithesis_random and logged), and the Conway genesis is baked into the image, so any timeline replays exactly. The local SanCov applyblock corpus + harness reproduce on the build host with --seed determinism.

Scope & honest limitations

  • A clean result is strong evidence of robustness on the exercised surface within the time/seed budget — it is not a proof of total absence of defects. The value is the breadth of timelines × seeds × (this run) fault schedules explored.
  • The Antithesis triage view summarises passing assertions by category (e.g. "Antithesis SDK 43/44 passed"); it does not render each dwarf_decoder_* applyblock assertion as an individual row. The bug-oracle verdict is unambiguous (no always(...) failure; node error/critical-free), but exact per-assertion pass counts would require the report's Logs-explorer drill-down.
  • No explicit post-fault recovery / CaughtUp / fork-depth liveness assertion is rendered in the summary; recovery under faults is evidenced by the run Completing and logging zero errors/criticals across the faulted duration, rather than by a dedicated liveness property.
  • The applyBlock corpus is a baked Conway set (varied but fixed); content diversity (multi-asset/mint, Plutus-script, governance-action branches) remains the natural next corpus extension. The live in-process volume is bounded per timeline; the 8h local SanCov soak (1.85M execs, 28k edges) is the high-volume coverage-guided complement.
